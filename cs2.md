2. CS2 醒来，恢复执行 `goto_to()`。
3. 此时列车已经在 B 地表了，CS2 直接给它写下前往 B 地表需求站的最后一段时刻表。

### 一、 CS2 的世界观：有向拓扑图与“路段”分割

CS2 的底层逻辑极其严密，完美契合了你单向传送门的物理特性，并规避了原生 Factorio 引擎“无法跨地表寻路”的死穴。

* **有向拓扑网 (Directed Topology)：** CS2 不会盲目假设地表是双向连通的。它会逐个地表向外发问：“从 A 能去哪？”。只要玩家建了 `A -> B` 的入口，你的模组就回答“能去 B”。CS2 据此画出单向路线图。如果玩家没建 `B -> A` 的回去路线，CS2 就会聪明地判定为“无路径”，从根源上掐断了有去无回的死锁。
* **按“路段 (Leg)”独立结算：** 一次完整的物流不被看作一个整体，而是被严格切分为三段：
1. 车库 -> 供货站 (`goto_from`)
2. 供货站 -> 需求站 (`goto_to`)
3. 需求站 -> 车库 (`complete`)



### 二、 核心交互机制：完美的“交接代管 (Handoff)”

在每一个路段发车前，CS2 都会与你的模组进行一次极其默契的“握手”。

1. **发车前的灵魂拷问 (拦截阶段)：**
在某个路段（例如装满货准备去需求站）落笔写时刻表的前一瞬间，CS2 发现目标在另一个星球。它立刻广播：“这段路跨界了，RiftRail 能接管吗？”
2. **全面放权 (挂机阶段)：**
你回答 `true`。CS2 瞬间停写时刻表，将列车标记为免监控状态 (`volatile`)，将自己的订单状态改为代管中 (`plugin_handoff`)，然后直接挂机休息。**此时，你拿到的是一辆装着货物、且你清楚知道其最终目的地的列车。**
3. **跨界穿梭 (你的主场)：**
你给列车写入临时时刻表，让它开进你的传送门，并在目标地表重新拼装出这辆列车。这个过程中，你那套防死锁、零内存分配的极致逻辑在黑盒内完美运转，CS2 一无所知。
4. **唤醒与二次发车 (归还阶段)：**
在新地表拼装完成后，你调用 CS2 的专属接口 `route_plugin_handoff`。CS2 瞬间苏醒，发现列车已经和目标车站在同一个星球了，于是它自信地拿起笔，给这辆拥有新 ID 的列车写下了最后一段原生时刻表。

### 三、 未来实战指南：当 CS2 完善后，你该怎么做？

如果半年或一年后，CS2 彻底发布了稳定版，并且你决定要在枯燥的业余时间里给玩家们一个巨大的惊喜，你的开发工作其实**极其轻松**，完全不需要改动 RiftRail 现有的核心流转逻辑。

你只需要在外部写一个非常薄的“接口翻译层”：

1. **注册为 CS2 的路由插件：**
调用 CS2 的 `remote` 接口，把 RiftRail 登记为 `route_plugin`。
2. **提供拓扑查线员 (`train_topology_callback`)：**
写一个小函数。当 CS2 问你“从地表 A 能去哪”时，你遍历一下 A 地表上现存的入口实体，提取出它们配对的出口所在的地表 ID，打包成一个列表扔给 CS2。
3. **提供路口拦截员 (`route_callback`)：**
写一个拦截函数。当 CS2 准备发车时，判断一下它传给你的 `train_home_surface` 和 `to_stop_entity.surface` 是否一致。如果不一致，返回 `true` 宣告接管。
4. **在出口增加一句话 (`route_plugin_handoff`)：**
在你现有的传送完成、列车重新生成的代码末尾，加一行 `remote.call("cs2", "route_plugin_handoff", delivery_id, new_luatrain)`。把焕然一新的火车还给它。
